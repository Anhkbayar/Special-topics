## 1. Interpreter хэрхэн ажилладаг вэ?

**Interpreter** нь эх кодыг бүхэлд нь урьдчилан машин код болгоод хадгалдаггүй. Харин кодыг:
1. уншина,
2. задлана,
3. тайлбарлана,
4. дарааллаар нь ажиллуулна.

Өөрөөр хэлбэл, програм ажиллах мөчид кодын утга, бүтэц, үйлдлийг шат дараатай ойлгож гүйцэтгэдэг.

Interpreter ажиллах ерөнхий үе шат:
- **Lexing / Tokenizing** – кодыг жижиг нэгжүүдэд хуваана
- **Parsing** – эдгээр нэгжээр syntax tree үүсгэнэ
- **Execution** – tree эсвэл intermediate representation-ийг ашиглаж ажиллуулна

Зарим interpreter нь эх кодыг шууд мөр мөрөөр нь ажиллуулдаг бол, зарим нь эхлээд **bytecode** болгон хөрвүүлээд дараа нь virtual machine дээр ажиллуулдаг.

## 2. AOT compiler болон JIT compiler хоорондын ялгаа нь юу вэ??

AOT compiler нь програмыг **ажиллуулахаас өмнө** бүхэлд нь эсвэл ихэнх хэсгийг машин код болгон хөрвүүлдэг.

### Жишээ:
- C
- C++
- Rust

### Давуу тал:
- Runtime үед хурдан
- Startup performance сайн
- Compiler optimization урьдчилан хийгдсэн байдаг

### Сул тал:
- Platform-specific binary үүсгэнэ
- Зарим runtime мэдээллийг ашиглаж optimize хийж чаддаггүй

---

## JIT (Just-In-Time) Compiler
JIT compiler нь програм **ажиллаж байх үед** кодын зарим хэсгийг машин код болгон хөрвүүлдэг.

### Давуу тал:
- Runtime behavior дээр үндэслэж optimize хийж чадна
- Hot path-уудыг маш сайн оновчилж чадна
- Dynamic language-д тохиромжтой

### Сул тал:
- Startup overhead нэмэгдэнэ
- Илүү төвөгтэй хэрэгжилт шаарддаг
- Memory илүү хэрэглэж магадгүй

## 3. CPython болон V8 нь аль төрлийн хөрвүүлэгч вэ?

### CPython
Cpython ний process бүрт global interpreter lock ажиллана. 
Cpython ий compiler нь abstract syntax tree үүсгэнэ. Түүний дараа bytecode үүсгэнэ. 
Эцэст нь python virtual machine дээр bytecode ийг ажиллуулна

### V8
V8 нь эхлээд parser ашиглан abstract syntax tree үүсгэнэ. Түүний дараа Ignition AST ийг Bytecode болгоно. Сүүлд нь TurboFan энэ bytecode ийг машины хэл рүү хөрвүүлнэ. Хөрвүүлэгдсэн код нь Runtime ийн үед optimization хийгдэнэ. 


## 4. Code smell ямар үед үүсдэг вэ? Түгээмэл code smell үүдийг дурд.
Code smell ихэвчлэн дараах нөхцөлд үүснэ:

- Хурдан дуусгах гэж яараад дизайн бодолгүй код бичих
- Давхардсан код олон газар үүсэх
- Нэг класс эсвэл функц хэт олон үүрэг авах
- Нэршил ойлгомжгүй байх
- Хэт урт method/class үүсэх
- Өмнөх кодыг ойлголгүй patch нэмж явах
- Refactor хийхгүйгээр feature нэмсээр байх
- Test хангалтгүй байх
- Architecture дүрэм зөрчих

---

### 1. Long Method
Хэт урт method. Ойлгох, test хийх, өөрчлөхөд хэцүү.

### 2. Duplicate Code
Ижил эсвэл маш төстэй код олон газар давтагдах.

### 3. Long Parameter List
Method дэндүү олон параметртэй байх.

### 4. God Object / God Class
Системийн хэт их логик нэг объект дээр төвлөрсөн байх.

### 5. Switch Statements Everywhere
Олон `if-else` эсвэл `switch` логик давтагдаж, polymorphism хэрэглэх боломжтой байхад ашиглаагүй байх.

## 5. Хэрхэн урьдчилан сэргийлэх вэ?
### 1. Clean code зарчим баримтлах
- Тодорхой нэршил ашиглах
- Жижиг, ойлгомжтой method бичих
- Нэг method нэг үндсэн үүрэгтэй байх

### 2. SOLID зарчим хэрэглэх
- **SRP** – нэг класс нэг хариуцлагатай байх
- **OCP** – өөрчлөхөөс илүү өргөтгөх байдлаар шийдэх
- **LSP, ISP, DIP** – dependency болон abstraction-ийг зөв ашиглах

### 3. Code review хийх
Бусад хүн харахад:
- үнэрүүдийг хурдан олно
- дизайн асуудал илүү тодорхой харагдана

### 4. Automated test бичих
Test сайн байвал refactor хийхэд айдас багасна. Ингэснээр муу бүтэцтэй кодыг засах боломж нэмэгдэнэ.

### 5. Static analysis tools ашиглах
Жишээ нь:
- SonarQube
- Checkstyle
- ESLint
